## Training history (for context)

```
v4  9-scan baseline                            F1 = 97.95%
v5  60-scan warm                               F1 = 97.26%  (stuck at v4 LR too low)
v6  60-scan fresh, rebalanced loss             F1 = 98.12%  [previous raw SoTA]
v7  99-scan fresh, v6 recipe                   F1 = 97.62%  (precision crash)
v8  MidUNet (1.86M params), fresh              F1 = 98.01%
v9  edge oversample + hard-neg + normed loss   F1 = 97.98%
v10 precision-biased loss (base=100, UP<OP)    COLLAPSED  (sigmoid → 0)
v11 warm from v6, 99 cleaned, 3e-4 LR          F1 = 97.89%
v12 FRESH random init, 99 cleaned, spatial UP  F1 = 98.30% raw / 99.46% + clf_v5
```

Key lessons:
- UP >= OP in loss must be maintained (v10 collapse lesson)
- Loss normalisation must be checked when changing anything (v9 crash)
- Data cleanup matters more than model changes at 98%+ F1 (label artifacts
  were propping up fake TP counts across several versions)

## Known issues / possible follow-ups

1. **Right-edge miss rate (77.78%, 9 pupae)** is actually higher than
   left-edge (36.81%) proportionally — but tiny sample, only 9 labels
   near right edge. Worth checking if there's a right-side scanner
   artifact we missed. Stephen has not labeled many right-edge pupae;
   possible data sparsity issue.

2. **Heldout eval hasn't been re-run on the latest clf_v5**. Earlier
   (on V11 + v4 with 79-train/20-heldout split) we saw V11 + clf_v4
   heldout F1 = 98.79%. Would be good to re-run with V12 + clf_v5 to
   confirm real generalisation rather than self-eval overfit.

3. **Center-cluster over-split** — 35 of 38 center misses are in dense
   pupa clusters (NN distance < 25 px). This is a fundamental detector
   limit and likely cannot be solved without a new architecture (e.g.,
   cluster-aware post-processing or a graph-based dedup pass).

4. **No automated labeling pipeline** yet. When Anthony says "seed a
   better set of 40 scans," someone needs to label them. The diff
   review tool works but requires manual time.
